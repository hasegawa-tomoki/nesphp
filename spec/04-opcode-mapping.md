# 04. Zend opcode → handler mapping

[← README](./README.md) | [← 03-vm-dispatch](./03-vm-dispatch.md) | [→ 05-toolchain](./05-toolchain.md)

Zend opcode numbers shift between PHP versions, so we **version-lock to PHP 8.4**. The exact numbers come from `php-src/Zend/zend_vm_opcodes.h` (8.4 branch) and are hardcoded.

## Versioning rules

- serializer.php and the VM (ca65) read the same constant table
- This file (`spec/04-opcode-mapping.md`) is the single source of truth for that table
- When PHP minor version changes, update the table here and rebuild both serializer and VM
- If `php_version_major/minor` in the op_array header isn't 8.4, the VM halts immediately

## Implemented Zend opcodes

Column "emit (host)" shows it is generated by the host-compile path (serializer.php). Column "emit (L3S)" shows it is emitted by the on-NES compiler (vm/compiler.s) ([13-compiler](./13-compiler.md)).

| Zend opcode | Number | emit (host) | emit (L3S) | nesphp behavior |
|-------------|------|:---:|:---:|----------------|
| `ZEND_NOP` | **0 (0x00)** | ✓ | — | Do nothing, advance PC (placeholder for intrinsic folding) |
| `ZEND_ADD` | **1 (0x01)** | ✓ | ✓ (P2) | op1 + op2 (IS_LONG assumed) → result. 16-bit signed addition |
| `ZEND_SUB` | **2 (0x02)** | ✓ | ✓ (P2) | op1 − op2 → result. 16-bit signed subtraction |
| `ZEND_SL` | **6 (0x06)** | ✓ | ✓ (S1) | op1 << op2 → result. 16-bit logical left shift |
| `ZEND_SR` | **7 (0x07)** | ✓ | ✓ (S1) | op1 >> op2 → result. 16-bit arithmetic right shift (sign-preserving) |
| `ZEND_BW_OR` | **9 (0x09)** | ✓ | ✓ (R3) | op1 \| op2 → result. 16-bit bitwise OR (IS_LONG) |
| `ZEND_BW_AND` | **10 (0x0A)** | ✓ | ✓ (R3) | op1 & op2 → result. 16-bit bitwise AND (IS_LONG) |
| `ZEND_IS_IDENTICAL` | **16 (0x10)** | ✓ | ✓ (P3) | Same type AND same value. Strings go through `values_equal_content` (len + val[] content compare) |
| `ZEND_IS_NOT_IDENTICAL` | **17 (0x11)** | ✓ | ✓ (P3) | Negation |
| `ZEND_IS_EQUAL` | **18 (0x12)** | ✓ | ✓ (P3) | Shares the `IS_IDENTICAL` implementation (PHP type juggling not supported) |
| `ZEND_IS_NOT_EQUAL` | **19 (0x13)** | ✓ | ✓ (P3) | Negation |
| `ZEND_IS_SMALLER` | **20 (0x14)** | ✓ | ✓ (P3) | op1 < op2 (IS_LONG signed 16-bit). Standard `BVC + EOR #$80 + BMI` idiom |
| `ZEND_ASSIGN` | **22 (0x16)** | ✓ | ✓ (P2) | op1 (IS_CV) ← op2 value. Direct copy of 4B tagged value |
| `ZEND_QM_ASSIGN` | **31 (0x1F)** | ✓ | — | op1 → result. Value copy |
| `ZEND_PRE_INC` | **34 (0x22)** | ✓ | ✓ (Q3) | `++$x`. Increment op1 (IS_CV), result = new value |
| `ZEND_PRE_DEC` | **35 (0x23)** | ✓ | ✓ (Q3) | `--$x`. Decrement op1, result = new value |
| `ZEND_POST_INC` | **36 (0x24)** | ✓ | ✓ (Q3) | `$x++`. Increment op1, result = old value |
| `ZEND_POST_DEC` | **37 (0x25)** | ✓ | ✓ (Q3) | `$x--`. Decrement op1, result = old value |
| `ZEND_INIT_ARRAY` | **71 (0x47)** | ✓ | ✓ (U1) | op1 = capacity (raw u16); result = TYPE_ARRAY zval for the new array. Allocates from **ARR_POOL (PRG-RAM bank 1, 8KB)** |
| `ZEND_ADD_ARRAY_ELEMENT` | **72 (0x48)** | ✓ | ✓ (U1) | op1 = array TMP, op2 = element. Append a 4B tagged zval at array's count, count++ (pool elements are 4B; 16B zvals are literal-only) |
| `ZEND_FETCH_DIM_R` | **81 (0x51)** | ✓ | ✓ (U1) | op1 = array, op2 = index (IS_LONG); result = the element narrowed to 4B tagged |
| `ZEND_COUNT` | **90 (0x5A)** | ✓ | ✓ (U1) | op1 = array; result = IS_LONG(count) |
| `ZEND_OP_DATA` | **138 (0x8A)** | ✓ | ✓ (V1) | No-op standalone. Carries the value op (read by ASSIGN_DIM etc. from the next op's op1) |
| `ZEND_ASSIGN_DIM` | **147 (0x93)** | ✓ | ✓ (V1) | op1 = array, op2 = key (IS_UNUSED appends). Value comes from the next ZEND_OP_DATA's op1. VM_PC advances +24 (consumes the 2-op sequence at ZOP_SIZE=12 × 2). count = max(count, slot+1) |
| `ZEND_JMP` | **42 (0x2A)** | ✓ | ✓ (P3) | Unconditional branch to op1.num (op_index) |
| `ZEND_JMPZ` | **43 (0x2B)** | ✓ | ✓ (P3) | If op1 is falsy, branch to op2.num |
| `ZEND_JMPNZ` | **44 (0x2C)** | ✓ | — | If op1 is truthy, branch to op2.num |
| `ZEND_RETURN` | **62 (0x3E)** | ✓ | ✓ (M-A') | Enable PPUMASK → infinite loop |
| `ZEND_ECHO` | **136 (0x88)** | ✓ | ✓ (M-A') | Output op1 to the PPU nametable |

## nesphp custom opcode range (0xE0-0xFF)

Zend uses 0-209, so we reserve `0xE0-0xFF` as nesphp's territory. They are all generated via the serializer's pattern folding.

| opcode | Number | Role |
|---|---|---|
| `NESPHP_FGETS` | **0xF0** | Wait for controller, write the pressed-button `button_str_X` (IS_STRING) into result |
| `NESPHP_NES_PUT` | **0xF1** | Write 1 character at nametable (x, y). Forced-blanking only |
| `NESPHP_NES_SPRITE` | **0xF2** | Update OAM[$idx] (0-63) y / tile / x. $idx is runtime-int OK; $tile must be a literal. Doesn't touch the attribute byte (use `NESPHP_NES_SPRITE_ATTR`). First call switches to sprite_mode |
| `NESPHP_NES_PUTS` | **0xF3** | Write a string literal at nametable (x, y). Forced-blanking only, no wrap |
| `NESPHP_NES_CLS` | **0xF4** | Fill nametable 0 ($2000-$23FF) with spaces and reset `PPU_CURSOR`. Forced-blanking only |
| `NESPHP_NES_CHR_SPR` | **0xF5** | Switch the sprite-side 4KB CHR set (0-3). Under SXROM CHR-RAM this means a 4KB bulk transfer from PRG_BANK1 to PPU $1000-$1FFF (~25 ms blackout). See [11-chr-banks](./11-chr-banks.md) |
| `NESPHP_NES_CHR_BG` | **0xF6** | Switch the BG-side 4KB CHR set (0-3). Bulk transfer from PRG_BANK1 to PPU $0000-$0FFF. Shares the chr_bulk_transfer subroutine |
| `NESPHP_NES_BG_COLOR` | **0xF7** | Set the universal background color ($3F00) to a NES color code (0x00-0x3F) |
| `NESPHP_NES_PALETTE` | **0xF8** | Set palette colors 1-3. 4 args (id, c1, c2, c3) consume all of op1/op2/result/extended_value |
| `NESPHP_NES_ATTR` | **0xF9** | Assign a palette index (0-3) to a 2×2 tile block in the attribute table. Read-modify-write through a 64B RAM shadow |
| `NESPHP_NES_SPRITE_ATTR` | **0xFC** | Set OAM[$idx]'s (0-63) attribute byte. bit 0-1=palette / bit 5=priority / bit 6=hflip / bit 7=vflip. Both args runtime-int OK |
| `NESPHP_NES_RAND` | **0xFD** | One step of a 16-bit Galois LFSR (tap = `$B400`, polynomial `x^16 + x^14 + x^13 + x^11 + 1`, period 65535). Returns IS_LONG (16-bit, written to the result slot). 0 args |
| `NESPHP_NES_SRAND` | **0xFE** | Overwrite the LFSR state with `$seed`. `$seed == 0` is the LFSR's degenerate state, so we substitute 1 |
| `NESPHP_NES_PUTINT` | **0xFF** | Write a 5-char right-justified unsigned int to nametable `(x, y)` in ASCII. Range 0..65535; leading zeros become ' ' (e.g. 99 → "   99"). All 3 args are runtime-int OK (reuses the result slot as the third input, like nes_sprite_at). Goes through the NMI sync queue while in sprite_mode |

### peek/poke (USER_RAM = $0700-$07FF, 256B)

A way to store / fetch byte-grained data without the 16-byte zval overhead. Reuses the post-compile-unused CV symbol table (`$0700-$07FF`) at runtime.

| opcode | Number | Role |
|---|---|---|
| `NESPHP_NES_PEEK` | **0xEC** | `nes_peek($offset)` → USER_RAM[$offset] returned as IS_LONG (byte). $offset is masked with $FF (8-bit wrap) |
| `NESPHP_NES_PEEK16` | **0xED** | `nes_peek16($offset)` → USER_RAM[$offset] \| (USER_RAM[$offset+1] << 8) as IS_LONG. Little-endian 16-bit |
| `NESPHP_NES_POKE` | **0xEE** | `nes_poke($offset, $byte)` → USER_RAM[$offset] = byte (low 1B only). No return value |
| `NESPHP_NES_POKESTR` | **0xEF** | `nes_pokestr($offset, $string)` → Bulk-copy raw string bytes to USER_RAM[$offset..]. Stops if $offset+len exceeds 256. Reuses the 3-arg intrinsic frame (op1=offset, result slot = string) |

**Example use**: Tetris keeps its 28-rotation shape table as a 56-byte string literal and bulk-loads it via `nes_pokestr(0, $shape_data)` at boot. Runtime restores the 16-bit shape with `nes_peek16($idx*2)` in one op. As an array it would cost 28 × 16 = 448 bytes; in USER_RAM it's just 56 bytes (8× more efficient).

### peek/poke_ext (USER_RAM_EXT = PRG-RAM bank 3, 8KB)

API that uses PRG-RAM bank 3 as **8 KB of generic byte storage**, enabled by the move to SXROM. For data tables / game-state buffers that don't fit in `nes_peek/poke`'s 256B internal RAM. 13-bit offsets (0-8191).

| opcode | Number | Role |
|---|---|---|
| `NESPHP_NES_PEEK_EXT` | **0xE8** | `nes_peek_ext($offset)` → reads 1 byte from `$6000+offset` in bank 3 |
| `NESPHP_NES_PEEK16_EXT` | **0xE9** | `nes_peek16_ext($offset)` → reads 2 LE bytes from bank 3 |
| `NESPHP_NES_POKE_EXT` | **0xEA** | `nes_poke_ext($offset, $byte)` → writes 1 byte to bank 3 |
| `NESPHP_NES_POKESTR_EXT` | **0xEB** | `nes_pokestr_ext($offset, $string)` → bulk-copy string bytes to bank 3 |

**Implementation note**: Each handler atomically switches MMC1 PRG-RAM to bank 3 on entry, performs the operation, and restores bank 0 on exit. Bank-switch cost is ~60 cycles per call (= MMC1 serial write × 2). `nes_pokestr_ext` cannot map STR_POOL (bank 2) and USER_RAM_EXT (bank 3) at the same time, so it does a 2-stage copy via the internal RAM text buffer ($0600-$06FF) (stage 1 = bank 2 → internal, stage 2 = internal → bank 3, max 255 bytes per chunk).

### Serializer pattern folding

The following Zend opcode sequences fold into a single custom opcode. Replacing them with `NOP`s preserves op_index, so subsequent JMP targets aren't broken.

**`fgets(STDIN)` → `NESPHP_FGETS`**
```
INIT_FCALL ... "fgets"          →  ZEND_NOP
FETCH_CONSTANT "STDIN"          →  ZEND_NOP
SEND_VAL T? ...                 →  ZEND_NOP
DO_ICALL                        →  NESPHP_FGETS (inherits the result slot)
```

**`nes_put($x, $y, "X" | $tile)` / `nes_puts($x, $y, "str")` → `NESPHP_NES_PUT` / `NESPHP_NES_PUTS`**
```
INIT_FCALL_BY_NAME ... "nes_put" →  ZEND_NOP
SEND_VAR_EX CV0 ... 1            →  ZEND_NOP  (record arg into pendingArgs)
SEND_VAR_EX CV1 ... 2            →  ZEND_NOP
SEND_VAL_EX string("X") ... 3    →  ZEND_NOP  (literal or runtime int; both work)
DO_FCALL_BY_NAME                 →  NESPHP_NES_PUT
                                    op1 = $x, op2 = $y, result = char/tile (3-arg runtime convention)
```

The third arg of `nes_put` may be **a 1-character string literal or a runtime int (tile number)**. The handler dispatches on TYPE_STRING / TYPE_LONG via `resolve_result`. `nes_puts` requires the third arg to be a `TYPE_STRING` literal; the VM streams len bytes from STR_POOL bank 2 to PPUDATA.

**`nes_sprite_at($idx, $x, $y, $tile)` → `NESPHP_NES_SPRITE`**
```
INIT_FCALL_BY_NAME 4 "nes_sprite_at" →  ZEND_NOP
SEND_VAL_EX $idx 1                  →  ZEND_NOP
SEND_VAL_EX $x 2                    →  ZEND_NOP
SEND_VAL_EX $y 3                    →  ZEND_NOP
SEND_VAL_EX int($tile) 4            →  ZEND_NOP
DO_FCALL_BY_NAME                     →  NESPHP_NES_SPRITE
                                        op1 = $idx, op2 = $x,
                                        result = $y, extended_value = $tile
```

A 4-arg intrinsic (like `nes_palette`) that reuses the result field as "third input". `$idx` / `$x` / `$y` accept runtime ints (CV/TMP/CONST); only `$tile` must be a literal. The VM clamps `$idx` with `& 0x3F` to convert to an OAM offset.

**`nes_sprite_attr($idx, $attr)` → `NESPHP_NES_SPRITE_ATTR`**
```
INIT_FCALL_BY_NAME 2 "nes_sprite_attr" →  ZEND_NOP
SEND_VAL_EX $idx 1                    →  ZEND_NOP
SEND_VAL_EX $attr 2                   →  ZEND_NOP
DO_FCALL_BY_NAME                       →  NESPHP_NES_SPRITE_ATTR
                                          op1 = $idx, op2 = $attr
```

Both args runtime-int OK. Writes the attribute byte (palette / flip / priority) directly to OAM[$idx*4+2].

**`nes_cls()` → `NESPHP_NES_CLS`**
```
INIT_FCALL_BY_NAME 0 "nes_cls"   →  ZEND_NOP
DO_FCALL_BY_NAME                 →  NESPHP_NES_CLS  (0 args)
```

**`nes_chr_bg(N)` / `nes_chr_spr(N)` → `NESPHP_NES_CHR_BG` / `NESPHP_NES_CHR_SPR`**
```
INIT_FCALL_BY_NAME 1 "nes_chr_bg"   →  ZEND_NOP
SEND_VAL_EX int(N) 1                →  ZEND_NOP
DO_FCALL_BY_NAME                    →  NESPHP_NES_CHR_BG
                                       op1 = int literal (IS_CONST)
```

The arg must be an integer literal at compile time (4KB bank number 0-7). BG and sprite are switchable independently (4KB CHR banking on MMC1).

**`nes_bg_color($c)` → `NESPHP_NES_BG_COLOR`**
```
INIT_FCALL_BY_NAME 1 "nes_bg_color"  →  ZEND_NOP
SEND_VAL_EX int($c) 1               →  ZEND_NOP
DO_FCALL_BY_NAME                     →  NESPHP_NES_BG_COLOR
                                        op1 = int literal (IS_CONST), NES color code 0x00-0x3F
```

The arg must be an integer literal (NES color code $00-$3F). Sets PPU $3F00 (universal background color).

**`nes_palette($id, $c1, $c2, $c3)` → `NESPHP_NES_PALETTE`**
```
INIT_FCALL_BY_NAME 4 "nes_palette"   →  ZEND_NOP
SEND_VAL_EX int($id) 1              →  ZEND_NOP  (pendingArgs[0])
SEND_VAL_EX int($c1) 2              →  ZEND_NOP  (pendingArgs[1])
SEND_VAL_EX int($c2) 3              →  ZEND_NOP  (pendingArgs[2])
SEND_VAL_EX int($c3) 4              →  ZEND_NOP  (pendingArgs[3])
DO_FCALL_BY_NAME                     →  NESPHP_NES_PALETTE
                                        op1 = $id, op2 = $c1,
                                        result = $c2, extended_value = $c3
```

**The first 4-arg intrinsic in nesphp**. Uses all four `zend_op` fields (op1, op2, result, extended_value) as inputs. Repurposing the result field as an "input" instead of an "output" deviates from Zend convention, but it's the only way to fold 4 args into a single instruction. All args must be integer literals at compile time. id 0-3 = BG palette, 4-7 = sprite palette.

**`nes_attr($x, $y, $pal)` → `NESPHP_NES_ATTR`**
```
INIT_FCALL_BY_NAME 3 "nes_attr"     →  ZEND_NOP
SEND_VAL_EX int($x) 1              →  ZEND_NOP
SEND_VAL_EX int($y) 2              →  ZEND_NOP
SEND_VAL_EX int($pal) 3            →  ZEND_NOP  (literal or runtime, both fine)
DO_FCALL_BY_NAME                    →  NESPHP_NES_ATTR
                                       op1 = $x, op2 = $y,
                                       result = $pal (palette index 0-3, 3-arg runtime convention)
```

x, y are coordinates of 2×2 tile (16×16 pixel) blocks (x: 0-15, y: 0-14). `$pal` is **runtime-int OK** (since commit 0670d6a, same "result-slot-as-third-input" pattern as `nes_putint` / `nes_sprite_at`). tetris.php uses this for per-piece dynamic palette assignment. The VM read-modify-writes a 64B RAM shadow (ATTR_SHADOW) and reflects it to the attribute table.

### Jump target encoding

In opcache dumps a jump target is shown as a **4-digit raw op_index** like `JMP 0005` / `JMPNZ T4 0002`. The serializer embeds it as a uint16 in the appropriate operand field's 2 bytes (`op1` for JMP, `op2` for JMPZ/JMPNZ) with operand type = `IS_UNUSED` (0x00). The VM converts to an absolute ROM address with `op_index * ZOP_SIZE + OPS_FIRST_OP` (= idx*12 + $6010) and stores it in `VM_PC`. The `jmp_compute_pc` helper centralizes the idx*12 calculation.

### Truthy / falsy (for JMPZ/JMPNZ)

| zval type | Verdict |
|---|---|
| `IS_NULL` / `IS_FALSE` / `IS_UNDEF` | falsy |
| `IS_TRUE` | truthy |
| `IS_LONG` | truthy iff value ≠ 0 |
| `IS_STRING` | always truthy (simplified; PHP's `""` / `"0"` are falsy but unsupported) |

The numbers come from `/opt/homebrew/Cellar/php/8.4.6/include/php/Zend/zend_vm_opcodes.h`.

### Operand slot encoding

The CV / TMP_VAR / VAR `op.var` field carries **slot number × 16** (close to Zend's runtime byte-offset convention, minus the `sizeof(zend_execute_data)` offset). The VM does `LSR / LSR` to multiply by 4, adds it to `VM_CVBASE` / `VM_TMPBASE`, and accesses the 4B tagged value slot.

The serializer translates `CV0($a)` / `T2` / `V1` from opcache dumps to `IS_CV/0*16`, `IS_TMP_VAR/2*16`, `IS_VAR/1*16` respectively.

## operand type (canonical values from `zend_compile.h`)

| Value | Name | Meaning |
|----|------|------|
| 0x00 | IS_UNUSED | Not used |
| 0x01 | IS_CONST | op*.constant is a byte offset into the literals array |
| 0x02 | IS_TMP_VAR | op*.var is a TMP slot number |
| 0x04 | IS_VAR | op*.var is a VAR slot number |
| 0x08 | IS_CV | op*.var is a CV (compiled variable) slot number |

> **Note**: an earlier draft of this spec listed wrong values like IS_CV=16, IS_UNUSED=8 (confused with another set of Zend flag constants). The values above are correct.

## Extended-goal opcodes

### Integer arithmetic / comparison

| Zend opcode | Role |
|-------------|------|
| `ZEND_ADD` | Addition. OP1+OP2 (both IS_LONG assumed), result written back to RESULT |
| `ZEND_SUB` | Subtraction |
| `ZEND_MUL` | Multiplication (16x16→16-bit routine on 6502, ~50 lines) |
| `ZEND_DIV` | Division (candidate for unsupported; the routine is heavy) |
| `ZEND_MOD` | Modulo (same) |
| `ZEND_IS_EQUAL` | Equality → IS_TRUE/FALSE |
| `ZEND_IS_NOT_EQUAL` | Inequality |
| `ZEND_IS_SMALLER` | `<` |
| `ZEND_IS_SMALLER_OR_EQUAL` | `<=` |

### Control flow

| Zend opcode | Role |
|-------------|------|
| `ZEND_JMP` | Unconditional branch. op1.jmp_offset is pre-resolved to a ROM op index |
| `ZEND_JMPZ` | Branch if OP1 is false |
| `ZEND_JMPNZ` | Branch if OP1 is true |
| `ZEND_JMPZNZ` | Multi-target branch (candidate for unsupported) |

**Resolving jmp_offset**: at Zend runtime `op1.jmp_offset` carries a `zend_op*` delta, but the serializer rewrites it to **a 0-based uint16 op index in the NES ROM**. The VM multiplies that index by `ZOP_SIZE` (=12) and sets `VM_PC` to `OPS_BASE + 16 + index*12`.

### Variable operations

| Zend opcode | Role |
|-------------|------|
| `ZEND_ASSIGN` | Assign op2 to CV[op1] |
| `ZEND_QM_ASSIGN` | Copy op1 to TMP[result] (ternary etc.) |
| `ZEND_FETCH_CONSTANT` | Fetch a PHP constant (unsupported in MVP; `true`/`false`/`null` could be specialized) |

### Strings

| Zend opcode | Role |
|-------------|------|
| `ZEND_CONCAT` | Concatenate OP1 and OP2. Single fixed RAM-buffer scheme (see CONCAT buffer in [06-display-io](./06-display-io.md)) |
| `ZEND_CAST` | Type cast (int → string etc.; unsupported in MVP) |

### Function calls (built-in specialization)

| Zend opcode | Role |
|-------------|------|
| `ZEND_INIT_FCALL` | Prepare a function call. op2.constant is the literal offset of the function name |
| `ZEND_SEND_VAL` | Push an arg |
| `ZEND_SEND_VAR` | Push an arg (variable) |
| `ZEND_DO_FCALL` | Execute the function |

### Pattern folding for built-ins

When PHP calls `fgets(STDIN)` or `nes_sprite_set(...)`, Zend emits

```
INIT_FCALL N "fname"
SEND_VAL/SEND_VAR ...
...
DO_FCALL
```

a **3-6 instruction sequence**. The serializer reads the function name literal from `INIT_FCALL` and folds it into a special built-in ID:

| Function name | Folded to |
|--------|----------|
| `fgets` (with `STDIN` arg) | `BUILTIN_READ_INPUT` |
| `nes_sprite_set` | `BUILTIN_SPRITE_SET` |
| `nes_sprite_move` | `BUILTIN_SPRITE_MOVE` |

In the `ZEND_DO_FCALL` handler, the VM dispatches on the built-in ID (resolved through a table inside the op_array) to read the controller / write to the OAM shadow. See [06-display-io](./06-display-io.md).

---

## Unsupported opcode handling

Any Zend opcode not listed above falls back to `handle_unimpl` (display `UNIMPL <hex>` on screen and halt). The serializer **aborts as a compile error** when it sees one (telling the user "this PHP construct isn't supported by nesphp").

Notable unsupported:

- `ZEND_NEW` / `ZEND_CLONE` (objects)
- `ZEND_FE_RESET_R` / `ZEND_FE_FETCH_R` (foreach)
- `ZEND_CATCH` / `ZEND_THROW` (exceptions)
- `ZEND_GENERATOR_*` (generators)
- `ZEND_FETCH_OBJ_R` / `ZEND_ASSIGN_OBJ` (object properties)
